# NovelBuilder 章节生成架构

生成链路统一为「证据包 -> 场景卡 -> 正文 -> 状态沉淀 -> 质量门」。

## 证据包

- 世界宪法、世界观、角色当前状态、项目级创作规则。
- 卷脉络、近期摘要、上一章结尾、语义检索片段、风格样本。
- 本章大纲、伏笔、术语表、字数和断章约束。

证据包只提供本章可用信息，避免模型自行提前透支后续剧情。

## 场景卡

每章内部拆成 1 到 3 张场景卡，不输出给读者。每张卡包含：

- 目标
- 障碍
- 信息增量
- 关系变化
- 承接证据
- 断章风险点

场景卡用于限制新增事件、角色、道具和未来卷信息。

## 正文生成

- 正文只展开本章大纲事件。
- 以对话、动作和选择推进。
- 环境描写必须改变人物判断、行动或关系压力。
- 章节结尾断在动作、对话或信息揭露高点，不写总结、展望和升华。

## 状态沉淀

生成后按顺序沉淀：

- 章节摘要
- 角色状态
- 伏笔状态
- 卷脉络
- 语义索引
- 质量报告

批量生成按章节顺序运行，让上一章沉淀结果进入下一章证据包。

## 质量门

质量门包含确定性检查和 LLM 审稿：

- 模板句
- 机械停顿
- 通用环境套话
- 生活细节偏置
- 句首代词单调
- 句长变化不足
- 情绪积极偏见
- 题材规则偏离

失败时把缺陷反馈写入下一轮生成上下文，不在数据库事务中等待 LLM。

## 参考项目吸收点

- 借鉴 `ainovel-cli` 的核心引擎、模型提供方、缓存、导出分层思路。
- 借鉴 `inkos` 的术语表、词汇疲劳、资源账本和连续章节约束。
- 避免参考仓库 issue 暴露的问题：章节走向失控、人称/POV 漂移、模型配置脆弱、导入内容污染上下文、空白页面缺少兜底。

## AI 指纹规则

- 避免“看了一眼、心跳漏了一拍、眼眶红了”三件套。
- 避免“不是……是……”固定辨析模板。
- 避免“沉默/安静了 X 秒/愣了一下”的机械停顿。
- 避免“空气中弥漫着 XX 气味”“只剩空调/显示器/机器嗡鸣”的通用环境套话。
- 现代中国场景不要默认咖啡、星巴克、糖醋排骨、炖排骨、烤肉店、日料店。
- 代词、句首、句长要有变化。
- 允许角色自然流露烦躁、疲惫、尴尬、厌恶、后悔等负面情绪，避免 AI 式积极偏见。
